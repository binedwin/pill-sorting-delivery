# 🏭 스마트 컨베이어 벨트 색상 분류 시스템 (Smart Conveyor Sorting System)

**작성자:** 이정빈 (SSAFY 임베디드 트랙)
**환경:** Raspberry Pi 5, Python 3.11

## 📌 프로젝트 개요
본 프로젝트는 라즈베리파이 5와 비전 AI(OpenCV)를 활용하여 컨베이어 벨트 위의 물체를 인식하고 지정된 색상에 따라 자동으로 경로를 분류하는 스마트 공정 자동화 시스템입니다. 
추후 로봇 팔(Dobot)과의 연동을 고려하여, 신호 수신부터 물체 이송, 비전 검사, 물리적 분류, 배출까지의 전체 시퀀스를 스레드(Thread)와 상태 기반(State-based)으로 정밀하게 제어합니다.

## 🛠️ 기술 스택 및 하드웨어
* **Hardware:** Raspberry Pi 5, USB Webcam, Servo Motor (분류기), Stepper Motor (컨베이어 벨트, L298N/Relay 연동)
* **Software:** Python 3, OpenCV (Vision AI), NumPy, `lgpio` (RP1 칩셋 하드웨어 제어)

## ⚙️ 시스템 동작 시퀀스 (Sequence Flow)
1. **신호 대기:** 로봇 팔(Dobot)이 컨베이어 벨트 위에 상자를 적재하고 신호를 송신. (현재는 터미널 `Enter` 입력으로 대체)
2. **1차 이송:** 신호 수신 시 스텝 모터가 가동되어 상자를 비전 카메라 하단(검사 위치)으로 2초간 이송 후 정지.
3. **비전 검사 (Vision Inspection):** 카메라 버퍼를 갱신한 후, HSV 색상 공간 변환 및 모폴로지 연산을 통해 객체의 색상(Red/Blue)을 판별.
4. **경로 분류:** 판별 결과와 타겟 색상의 일치 여부에 따라 서보모터가 지정된 각도(좌 95도 / 우 175도)로 이동하여 물리적 차단벽 형성.
5. **2차 이송 (배출):** 컨베이어 벨트를 2초간 재가동하여 상자를 분류된 경로로 배출.
6. **초기화:** 분류기가 중앙(135도)으로 복귀하며 다음 사이클 대기.

### 🚨 트러블 슈팅 (Troubleshooting) 

개발 과정에서 직면했던 라즈베리파이 하드웨어 제어 및 소프트웨어 충돌 문제와 그 해결 과정을 기록합니다.

#### 1. 서보모터의 미세 떨림(Jitter) 및 각도 오차 발생
* **현상:** OpenCV 비전 처리가 돌아가는 동안 서보모터가 지정된 각도(175도)에 도달하지 못하고 140도 부근에서 멈추거나 1도씩 미세하게 계속 떨리는 현상 발생. `gpiozero` 라이브러리 사용 시 증상 심화.
* **원인:** 1. 영상 처리 연산으로 인해 CPU 점유율이 높아져 소프트웨어 PWM 신호의 타이밍(Pulse Width)이 밀림.
  2. 하이레벨 라이브러리(`gpiozero`)의 내부 펄스 폭 계산식과 실제 사용 중인 보급형 서보모터의 물리적 인식 스펙이 불일치.
* **해결 방안:** * 모터 특성에 맞춘 정확한 펄스 폭 계산식 `(angle / 270) * (0.0025 - 0.0005) + 0.0005`을 직접 구현.
  * 서보모터가 작동하는 0.4초(`for _ in range(20):`) 동안은 비전 연산을 잠시 홀딩하고 CPU가 PWM 신호 생성에만 100% 집중하도록 코드를 재설계하여 떨림 현상 완벽 제거.

#### 2. 라즈베리파이 5 환경에서의 라이브러리 호환성 및 충돌 문제
* **현상 1:** 하드웨어 PWM을 위해 `pigpio` 데몬을 실행했으나 `unknown rev code` 에러 발생 및 구동 실패.
* **현상 2:** 기존에 정상 작동하던 `gpiod` 버튼/모터 코드가 가상환경에서 `AttributeError: module 'gpiod' has no attribute 'Chip'` 에러를 발생시킴.
* **원인:**
  1. `pigpio`는 과거 Broadcom 아키텍처에 종속되어 있어 라즈베리파이 5의 새로운 자체 실리콘 칩인 **'RP1'**을 물리적으로 지원하지 않음.
  2. 가상환경 내에 설치된 `gpiod` 버전이 최신(v2.x)으로 업데이트되면서 기존(v1.x)의 `Chip` 문법이 Deprecated 됨.
* **해결 방안:**
  * 버전 의존성 문제와 하드웨어 비호환 문제를 동시에 해결하기 위해, 라즈베리파이 5의 공식 로우레벨 제어 라이브러리인 **`lgpio`**로 핀 제어 시스템을 전면 교체 및 단일화.

#### 3. 카메라 버퍼 딜레이로 인한 인식 오류
* **현상:** 컨베이어 벨트 이송 후 비전 인식을 시작할 때, 현재 상자가 아닌 과거의 빈 프레임을 인식하여 객체를 찾지 못하는 문제 발생.
* **원인:** `cv2.VideoCapture`의 내부 버퍼에 1차 이송(2초) 동안 녹화된 과거 프레임들이 쌓여 있어, `cap.read()` 호출 시 최신 프레임이 즉각 반영되지 않음.
* **해결 방안:** 비전 인식 로직 진입 직전 `for _ in range(5): cap.read()`를 수행하여 과거 버퍼를 강제로 비우고 가장 최신의 프레임을 가져오도록 동기화 처리.

#### 4. 스텝 모터(컨베이어 벨트) 비동기 제어
* **현상:** 컨베이어 벨트 구동 로직 실행 시 프로그램이 멈추거나 모터가 돌지 않음.
* **원인:** 일반 DC 모터와 달리 스텝 모터는 0.0001초 간격으로 지속적인 펄스(Step)를 주어야 구동됨. 이를 메인 루프에서 처리할 경우 비전 인식 코드가 블로킹(Blocking)됨.
* **해결 방안:** 파이썬의 `threading` 라이브러리를 도입하여 스텝 모터의 펄스 생성(가동)을 백그라운드 데몬 스레드로 분리. 메인 스레드는 비전 인식과 시퀀스(플래그 변경)에만 집중하도록 구조 개선.
